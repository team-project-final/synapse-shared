### 🔴 미추적 잔여 — 사실상 1개 클러스터 (W2 Kafka 보안)
1. **SASL/IAM vs TLS-only 인증 모델 결정 미종결** — W4_DAY1 실측: MSK가 SASL/IAM 미활성인데 KAFKA_AUTH_MATRIX는 IAM 전제 → "결정 필요"로 정지. 서비스는 TLS(security.protocol=SSL)만 배선(#54/#45/#48).
2. **Kafka ACL(토픽별 권한)** — SASL 부재 시 주체 없음 → 1번과 한 묶음.
3. **민감 토픽 접근 제한** — ACL 의존 → 1·2와 동일.
4. **TLS 전송 실측 검증**(openssl s_client 등) — SSL 배선됐으나 "전송 암호화 확인" 증거 문서 부재. MSK ACTIVE라 즉시 확보 가능.
5. **(W4) Step 11.1 staging 검증 시나리오 정의** — verify-argocd-deploy로 사실상 대체, 경미.

### 발표 리스크 결론
- 🔴 높음: **없음** (Kafka 보안은 기능 데모 표면에 안 드러남).
- 🟡 중간: TLS 전송 검증(4) — 보안 클레임 근거 공백. MSK ACTIVE라 5분 내 확보 권고.
- ⚪ 낮음: ACL/SASL 결정(1·2·3) — 명시적 종결로 닫힘, 구현 불필요.

## 5. 권고 (리포트 §5)

1. **TLS 전송 검증 1회 실행** → 증거 캡처 → 🟡 닫기. (MSK ACTIVE 활용; 실행 여부는 구현 플랜에서 결정)
2. **인증 모델 TLS-only 종결 선언** → KAFKA_AUTH_MATRIX(또는 D-002 계열)에 결정 1줄: "dev/staging은 TLS-only 채택. SASL/IAM·토픽 ACL은 발표 범위 외(향후 과제)." → 🔴 1·2·3을 ✅로 종결.
3. **나머지는 정합 표기만** — ✅해소는 근거 링크, 🔄는 W5_PLAN 연결.
