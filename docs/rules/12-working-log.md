# 12. 작업 로그 RULE — Working Log

> 기록 없는 작업은 안 한 거랑 같아. 커밋, PR, 일일 로그 전부 규칙대로.

---

## 12.1 커밋 메시지 [MUST]

**Conventional Commits** 형식을 따라.

```
<type>(<scope>): <subject>

[body — "왜" 이 변경이 필요한지] [SHOULD]

[footer — Breaking Change, Issue ref]
```

### 허용 type

| type | 용도 |
|------|------|
| feat | 새 기능 |
| fix | 버그 수정 |
| docs | 문서만 변경 |
| style | 포맷팅 (코드 의미 변경 없음) |
| refactor | 리팩토링 |
| test | 테스트 추가/수정 |
| chore | 빌드, CI, 의존성 등 |

### ✅ Good

```
feat(auth): 소셜 로그인 카카오 연동 추가

카카오 OAuth2 플로우를 구현하여 회원가입 없이 로그인 가능하게 함.
기존 이메일 가입자와 계정 병합 로직 포함.

Closes #42
```

```
fix(order): 할인 금액이 음수가 되는 버그 수정

할인율 > 100%인 쿠폰이 적용될 때 주문 금액이 음수로 계산되던 문제.
최소 결제 금액을 0원으로 하한 처리.
```

### ❌ Bad

```
수정함
```

```
fix bug
```

```
feat: 여러가지 고침 (로그인, 회원가입, 결제 전부)
```

> **이유**: 커밋 로그는 프로젝트의 역사야. "왜" 바꿨는지 적어야 6개월 후에 git blame 할 때 살아남아.

## 12.2 PR 본문 [MUST]

모든 PR에 아래 두 섹션 필수:

- `## 변경 사항` — 뭘 바꿨는지
- `## 테스트 결과` — 어떻게 검증했는지

PR 크기는 **400줄 이하** [SHOULD].

### PR 본문 템플릿

```markdown
## 변경 사항

- [feat] 카카오 소셜 로그인 연동
- [refactor] AuthService에서 OAuth 전략 패턴 분리

## 테스트 결과

- [x] KakaoOAuthServiceTest 전체 통과 (12 cases)
- [x] AuthE2E 시나리오 테스트 통과
- [x] 로컬에서 카카오 로그인 플로우 수동 검증 완료

## 관련 이슈

- Closes #42
- Related #38

## 스크린샷 (선택)

(해당 시 첨부)
```

### ✅ Good

- 변경 사항이 bullet으로 명확히 나열됨
- 테스트 결과에 구체적 수치/증거 있음
- 400줄 이내로 리뷰어가 30분 안에 볼 수 있음

### ❌ Bad

- PR 설명란 비어있음
- "잘 됩니다" 한 줄로 테스트 결과 대체
- 1000줄짜리 PR에 "전체적으로 리팩토링했습니다"

> **이유**: PR 본문은 리뷰어를 위한 배려야. 맥락 없이 코드만 던지면 리뷰 품질이 떨어지고 시간도 낭비돼.

## 12.3 HISTORY 일일 로그 [MUST]

매일 퇴근 전 `HISTORY.md` 파일을 갱신해. 3줄 형식:

```
한 일 / 이슈 / 내일 계획
```

### 형식

```markdown
## 2026-05-12 (월) — 홍길동

**한 일**
- feat(auth): 카카오 소셜 로그인 구현 완료
- test(auth): KakaoOAuthService 단위 테스트 12건 작성

**이슈**
- 카카오 API rate limit 때문에 E2E 테스트 간헐적 실패 → retry 로직 검토 필요

**내일 계획**
- 네이버 소셜 로그인 연동 시작
- rate limit 대응 방안 기술 검토 후 팀 공유
```

### ✅ Good

```markdown
## 2026-05-12 (월) — 김개발

**한 일**
- fix(order): 할인 음수 버그 수정 + 테스트 추가
- docs: API 명세서 할인 엔드포인트 업데이트

**이슈**
- 없음

**내일 계획**
- 쿠폰 만료 스케줄러 구현 착수
```

### ❌ Bad

```markdown
## 2026-05-12

오늘 좀 했음. 내일도 할 예정.
```

> **이유**: 일일 로그는 팀 전체의 진행 상황 파악용이야. 스탠드업 미팅 대신 비동기로 공유하는 거니까, 구체적이어야 의미가 있어.

## 요약 체크리스트

| 항목 | 레벨 | 핵심 |
|------|------|------|
| Conventional Commits | [MUST] | type(scope): subject |
| 커밋 본문에 "왜" | [SHOULD] | body에 변경 이유 작성 |
| PR `## 변경 사항` | [MUST] | bullet 나열 |
| PR `## 테스트 결과` | [MUST] | 구체적 검증 증거 |
| PR 400줄 이하 | [SHOULD] | 넘으면 분할 |
| HISTORY 매일 갱신 | [MUST] | 한 일 / 이슈 / 내일 |
