# 14. Task 문서 구조 RULE — Task Structure

> Task 문서는 팀의 공용 작업 지시서야. 형식이 통일돼야 누가 봐도 바로 착수 가능해.

---

## 14.1 필수 10필드 [MUST]

모든 Task 문서는 아래 10개 필드를 반드시 포함해야 해.

| # | 필드명 | 설명 | 예시 |
|---|--------|------|------|
| 1 | Task ID | 고유 식별자 | `TASK-007` |
| 2 | Title | 한 줄 요약 | 카카오 소셜 로그인 구현 |
| 3 | Owner | 담당자 | 홍길동 |
| 4 | Status | 진행 상태 | `TODO` / `IN_PROGRESS` / `DONE` |
| 5 | Priority | 우선순위 | `P0` (긴급) ~ `P3` (낮음) |
| 6 | Step Goal | 측정 가능한 목표 문장 | (14.2 참고) |
| 7 | Done When | 완료 조건 | (14.3 참고) |
| 8 | Scope | 범위 정의 | (14.4 참고) |
| 9 | Dependencies | 선행 Task | `TASK-005`, `TASK-006` |
| 10 | Due Date | 마감일 | `2026-05-16` |

### ✅ Good

- 10필드 전부 채워져 있고, 각 필드가 구체적

### ❌ Bad

- Step Goal 누락, Done When 없이 "적당히 완성되면"
- Status가 2주째 `IN_PROGRESS`인데 아무 업데이트 없음

> **이유**: 필드가 빠지면 "이거 끝난 거야 아닌 거야?" 같은 불필요한 커뮤니케이션이 생겨. 10필드면 그런 질문이 사라져.

---

## 14.2 Step Goal [MUST]

측정 가능한 문장으로 작성. 패턴:

```
[주체]가 [대상]에 [행위]를 [결과]한다
```

| | 예시 | 판정 |
|---|------|------|
| ✅ Good | 사용자가 카카오 계정으로 로그인하면 JWT 토큰을 발급받는다 | 측정 가능 |
| ✅ Good | CI 파이프라인이 PR 생성 시 Jacoco 커버리지 70% 미만이면 빌드를 실패시킨다 | 측정 가능 |
| ✅ Good | 관리자가 대시보드에서 일별 주문 통계를 차트로 조회한다 | 측정 가능 |
| ❌ Bad | 로그인 기능 개선 | 뭘 어떻게? 측정 불가 |
| ❌ Bad | 코드 품질 향상 | 기준 없음 |
| ❌ Bad | 성능 최적화 | 구체적 수치 없음 |

> **이유**: "개선", "향상", "최적화" 같은 단어는 완료 기준이 없어서 영원히 안 끝나. 주체+대상+행위+결과가 있어야 Done 판단이 가능해.

---

## 14.3 Done When 배치 [MUST]

Done When은 **Step Goal 바로 다음**에 고정 배치해. 다른 섹션 사이에 끼면 안 돼.

```markdown
## Step Goal
사용자가 카카오 계정으로 로그인하면 JWT 토큰을 발급받는다.

## Done When
- [ ] 카카오 OAuth2 인증 플로우 정상 동작
- [ ] 기존 이메일 계정과 카카오 계정 병합 처리
- [ ] KakaoOAuthService 단위 테스트 통과
- [ ] AuthE2E 카카오 로그인 시나리오 통과
- [ ] API 명세서 업데이트 완료
```

### ✅ Good

- Goal → Done When 순서 고정
- Done When 항목이 체크리스트(- [ ])로 작성돼 진행률 추적 가능

### ❌ Bad

```markdown
## Step Goal
로그인 구현

## Scope
(...)

## Dependencies
(...)

## Done When   ← Goal에서 너무 멀리 떨어짐
- 잘 되면 끝
```

> **이유**: Goal과 Done When은 "목표 ↔ 완료 조건" 쌍이야. 떨어뜨리면 읽는 사람이 왔다 갔다 해야 해.

---

## 14.4 Scope 구조 [MUST]

In Scope / Out of Scope 두 블록으로 고정. 이 형식 벗어나지 마.

```markdown
## Scope

### In Scope
- 카카오 OAuth2 연동 (Authorization Code Grant)
- 기존 이메일 계정과 소셜 계정 병합
- JWT 토큰 발급 및 리프레시

### Out of Scope
- 네이버/구글 소셜 로그인 (TASK-008에서 처리)
- 소셜 로그인 탈퇴 플로우 (TASK-012에서 처리)
- 관리자용 소셜 로그인 통계 대시보드
```

### ✅ Good

- In/Out이 명확히 분리됨
- Out of Scope에 "어디서 하는지" 레퍼런스 포함

### ❌ Bad

```markdown
## Scope
카카오 로그인 관련된 거 전부
```

> **이유**: Out of Scope가 없으면 "이것도 해야 되는 거 아냐?" 스코프 크립이 시작돼. 명시적으로 "안 한다"를 적어야 경계가 잡혀.

## 14.5 10단계 개발 워크플로 [SHOULD]

Task 착수부터 완료까지 아래 10단계를 따라.

| 단계 | 이름 | 핵심 활동 |
|------|------|-----------|
| ① | 요구사항 확인 | Task 문서 10필드 검토, 모호한 부분 질문 |
| ② | 설계 | API 스펙, 시퀀스 다이어그램, DB 스키마 초안 |
| ③ | 브랜치 생성 | `feat/TASK-007-kakao-login` 형식 |
| ④ | TDD 사이클 | 테스트 먼저 → 구현 → 리팩토링 |
| ⑤ | Security 1차 | 구현 중 보안 체크 (아래 체크리스트) |
| ⑥ | 자체 리뷰 | diff 전체 읽기, 불필요 코드 제거 |
| ⑦ | PR 생성 | 12-working-log 룰 준수 |
| ⑧ | 코드 리뷰 | 피드백 반영, 24h 이내 대응 |
| ⑨ | Security 2차 | 머지 전 최종 보안 점검 |
| ⑩ | 머지 & 배포 | main 머지 후 Done When 체크리스트 완료 처리 |

### Security 1차 체크리스트 (구현 중)

- [ ] SQL Injection 방어 — PreparedStatement / ORM 파라미터 바인딩
- [ ] XSS 방어 — 사용자 입력 이스케이프 처리
- [ ] 인증/인가 — 엔드포인트별 권한 확인
- [ ] 민감 데이터 — 비밀번호 해싱, 토큰 만료 설정
- [ ] 로깅 — 개인정보 마스킹 (이메일, 전화번호)

### Security 2차 체크리스트 (머지 전)

- [ ] 의존성 취약점 — `./gradlew dependencyCheckAnalyze` 또는 `npm audit`
- [ ] 시크릿 노출 — `.env`, API Key가 코드에 하드코딩되지 않았는지
- [ ] CORS 설정 — 허용 도메인 최소화
- [ ] Rate Limiting — 인증 관련 API에 적용 여부
- [ ] 에러 응답 — 스택 트레이스가 클라이언트에 노출되지 않는지

### ✅ Good

- 10단계 순서대로 진행하며 각 단계 산출물 남김
- Security 체크리스트 전부 체크 후 머지

### ❌ Bad

- 설계 없이 바로 코딩 시작
- Security 체크 건너뛰고 "나중에 하자"
- PR 올리고 리뷰 안 기다리고 셀프 머지

> **이유**: 워크플로가 없으면 각자 스타일대로 해서 품질 편차가 커져. 10단계는 최소한의 가드레일이야.
